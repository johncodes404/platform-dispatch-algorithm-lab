# 从一段代码理解“算法治理”

## 1. 算法不是一个人写出来的

现实企业里，程序员通常只完成最后一步。完整链条大致是：

```text
管理层确定经营目标
→ 产品与运营定义指标、考核和业务规则
→ 数据团队训练 ETA、出餐时间、超时风险等预测模型
→ 运筹团队设计匹配目标和约束
→ 工程团队把模型接入实时派单系统
→ 运营团队根据实验结果继续调整权重
```

因此，“谁写算法”至少有两种答案：代码由技术人员写，算法所服从的利益目标则由组织权力结构决定。

## 2. 本项目的五层结构

### 第一层：输入数据

订单包含商家位置、顾客位置、承诺时限、剩余出餐时间和楼宇交付时间；骑手包含位置、手头订单、路线重合度、容量、疲劳度、近期收入和在线状态。

现实中的难点是，这些数据并不天然客观。GPS 会漂移，商家可能晚出餐，电梯可能拥堵，系统也很难准确知道骑手是否疲劳。

### 第二层：预测

`predict_features()` 估计“如果把订单派给这个人，会发生什么”。真实平台一般会训练多个机器学习模型，但预测永远带有误差。

### 第三层：硬约束

`evaluate_candidates()` 先剔除离线、满载和超过疲劳红线的骑手。硬约束非常重要，因为只把安全写成一个低权重指标，意味着系统仍可以为了效率牺牲安全。

### 第四层：目标函数

不同量纲不能直接相加，所以程序先将每个指标归一化到 0～1，再乘以政策权重。权重就是最直观的制度选择。

### 第五层：执行和反馈

`dispatch()` 选择综合代价最低的人。真实系统还会记录骑手接单、拒单、轨迹和送达结果，再拿这些数据训练下一轮模型，形成反馈循环。

## 3. 不公平通常从哪里产生

1. **目标缺项**：只算履约效率，不算劳动者疲劳和收入稳定；
2. **风险转嫁**：商家出餐、交通、门禁等不确定性，最后由骑手超时承担；
3. **历史偏差循环**：历史上拿单多的人数据更好、预测更稳定，于是未来继续优先拿单；
4. **平均数压迫个体**：模型知道某栋楼平均需要五分钟，却不知道今天电梯坏了；
5. **权重不可见**：劳动者只看到结果，不知道为什么自己没单、降级或被处罚；
6. **指标逐步收紧**：效率提高后，新纪录可能变成下一轮考核基准，技术进步没有转化为休息，而是转化为更高劳动强度。

最后一点尤其值得注意：算法最强的地方不是它会“命令”劳动者，而是它能把管理变成连续、实时、低成本的微观控制。过去一个站长只能粗略管理几十人；现在系统可以逐秒观察位置、速度、接单与停留，并自动奖惩。这就是平台劳动中技术权力的实质。

## 4. 怎样把治理要求写回代码

治理不能只要求企业笼统承诺“算法向善”，而要进入软件可以执行的层面：

- 将最长连续工作时间写成硬约束；
- 将商家晚出餐和异常天气从骑手责任中剥离；
- 公示影响派单、计价、奖惩的主要因素；
- 记录每次自动决策的原因，允许骑手查询和申诉；
- 分别监测不同区域、用工类型和收入组的获单率与时薪；
- 算法更新前进行劳动影响评估，上线后保留回滚机制；
- 让劳动者代表参与目标与权重的讨论，而不只是被告知结果。

一句话概括：**算法治理不是让公众阅读几百万行源码，而是让目标、约束、数据、结果和责任能够被追问。**
