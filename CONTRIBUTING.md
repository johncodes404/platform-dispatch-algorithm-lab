# 参与贡献

感谢你愿意一起建设“平台算法权力实验室”。这个项目需要程序员，也需要教师、学生、骑手、研究者、设计师和普通公众。

## 可以贡献什么

- 教学：课程活动、年龄分层材料、复盘问题；
- 内容：角色、事件、情境与术语解释；
- 研究：可靠资料、数据出处、模型局限性说明；
- 设计：交互、视觉、无障碍和移动端体验；
- 开发：算法、前端、测试、性能和部署；
- 反馈：错误报告、课堂观察、学习者困惑与建议。

涉及劳动者经验时，请尊重当事人，不提交可识别个人的信息。研究资料应附来源，个人观察应明确标注为观察而非普遍事实。

## 开始之前

1. 阅读 README.md、AGENTS.md 和相关 docs 文档。
2. 搜索现有 Issue，确认没有重复讨论。
3. 小修复可直接提交 PR；新角色、核心规则、依赖或架构改动请先创建 Issue。
4. 在 Issue 中先写清楚“它解决什么学习问题”，再讨论技术实现。

## 推荐分支命名

- feature/简短主题
- fix/简短主题
- docs/简短主题
- test/简短主题
- research/简短主题

提交信息推荐使用：

- feat: 新功能
- fix: 修复问题
- docs: 文档变化
- test: 测试变化
- refactor: 不改变行为的重构
- chore: 工程维护

## Pull Request 要求

一个 PR 尽量只解决一个主题，并包含：

- 问题背景与学习目标；
- 主要改动；
- 页面变化的截图或录屏（如适用）；
- 验证方式和结果；
- 新增参数、数据或资料的来源；
- 已知限制和后续事项。

提交前请完成：

- [ ] 核心算法含义和界面解释保持一致
- [ ] Python 测试通过
- [ ] 桌面端和移动端主要流程可用
- [ ] 键盘可完成主要交互
- [ ] 没有提交个人数据、密钥或构建垃圾
- [ ] 相关 README/docs 已更新
- [ ] 新增复杂规则有说明“为什么”的注释

## 评审原则

评审优先看教学准确性、可解释性、尊重、可访问性和可维护性。评审针对方案和代码，不针对个人。对于价值冲突，记录不同方案的受益者、成本承担者和可逆性，再作决定。

核心模型或课堂目标的重大变化，建议至少由一名教学/研究贡献者和一名开发贡献者共同评审。
