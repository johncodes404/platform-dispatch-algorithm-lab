# 团队合作开发说明

## 1. 协作目标

团队不是只把功能“做出来”，还要保证每个机制能被学习者理解、被教师使用、被贡献者验证。每项工作都从一个学习问题开始，以可复现的验证和可讨论的文档结束。

## 2. 推荐角色

小团队可以一人兼任多个角色。

| 角色 | 主要责任 | 关键评审问题 |
| --- | --- | --- |
| 维护者 | 范围、版本、合并与社区秩序 | 是否符合使命，是否可持续维护 |
| 教学设计 | 学习目标、课堂流程、年龄适配 | 学习者能否观察并表达关键概念 |
| 算法/研究 | 模型、资料、假设与局限 | 指标是否可解释，证据是否可靠 |
| 前端/交互 | 页面、状态、可访问性 | 操作与反馈是否清晰一致 |
| 游戏设计 | 角色、回合、资源与平衡 | 机制是否产生有意义的取舍 |
| 测试/课堂观察 | 回归测试、试讲和问题记录 | 结果能否复现，哪里会被误解 |

## 3. 标准工作流

### 第一步：提出 Issue

Issue 至少写清：

- 面向谁；
- 想解决什么学习或使用问题；
- 当前证据或观察；
- 最小可行改动；
- 如何判断有效；
- 风险和待讨论问题。

核心规则、新依赖、新角色、用户数据或架构调整必须先讨论。错别字、小型无障碍修复和明确 Bug 可直接提交 PR。

### 第二步：确定范围

维护者、教学和实现贡献者确认：

- 本次做什么；
- 明确不做什么；
- 涉及哪些文件和概念；
- 是否需要资料来源、测试或课堂试用；
- 谁负责最终验收。

### 第三步：实现

- 从 main 创建短生命周期分支；
- 一次 PR 只处理一个主题；
- 先保留可工作的最小版本，再迭代体验；
- 核心规则写“为什么”的注释；
- 算法、解释、测试和文档同步更新。

### 第四步：双重评审

涉及核心模型或教学内容时，建议采用双重评审：

1. 技术评审：行为、边界、可访问性和维护成本；
2. 教学/研究评审：概念、证据、角色尊重和可能误导。

评审意见分为“合并前必须解决”“建议改进”“后续 Issue”，减少无休止扩张范围。

### 第五步：验证与发布

按完成标准验证，合并到 main 后检查 GitHub Pages。重要版本建立 Release，并记录面向教师和学习者的变化。

## 4. 完成标准

一项功能只有同时满足以下条件才算完成：

- [ ] 学习目标和用户场景明确
- [ ] 核心流程可操作，空状态和边界状态可理解
- [ ] 结果解释来自实际计算
- [ ] 相关自动化测试通过
- [ ] 桌面、窄屏、键盘主要流程已检查
- [ ] 不收集不必要的个人信息
- [ ] 资料来源、假设和局限已记录
- [ ] README 或专题文档已更新
- [ ] PR 说明验证方法和已知限制
- [ ] 发布页面已抽查

## 5. 决策原则

发生分歧时，用以下顺序判断：

1. 是否保护学习者和参与者；
2. 是否忠于教学目标和可核验事实；
3. 是否让受益、成本和风险承担者更可见；
4. 是否保留多种立场和复盘空间；
5. 是否能以较低维护成本持续运行；
6. 是否容易撤回或修正。

不能达成一致时，先做可逆的小实验，并记录假设和观察，不用技术实现掩盖价值争议。

## 6. 建议节奏

- 每周：异步更新 Issue，写清完成、下一步和阻塞；
- 每两周：选择一个可演示的小里程碑；
- 每个版本：至少一次非开发者试用；
- 每个学期：收集课堂反馈，复核术语、角色和难度；
- 每年：检查依赖、资料链接、隐私和社区规则。

## 7. 沟通约定

- 重要决定留在 Issue 或 PR，不只存在于私聊；
- 结论附上下文和理由，方便后来者接手；
- 不要求所有贡献者熟悉全部学科；
- 对劳动者经验和未成年人反馈提高隐私保护；
- 不把“算法算出来的”当作结束讨论的理由。

分支、提交和 PR 的具体格式见 CONTRIBUTING.md。
