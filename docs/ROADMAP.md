# 下一步开发计划

路线图围绕一条主线推进：先把单人互动实验打磨成可靠教学工具，再扩展为多角色课堂游戏。版本号表达能力阶段，不承诺固定日期；实际优先级由 Issue、课堂反馈和维护能力共同决定。

## 当前：v0.1 互动实验室

已具备：

- 单次派单的权重调整和候选人评分；
- 平台效率与劳动友好政策对比；
- 疲劳硬约束和多种突发情境；
- 六轮高峰连续模拟；
- 决策透视、Python 模型、测试与 GitHub Pages；
- 面向人和 AI 的开发、协作与教学文档。

完成判断：公开页面可访问，主要流程可在无账号、无后端的环境中使用。

## 下一步：v0.2 教学工具包

目标：让教师不需要解释代码，也能稳定组织一节课。

计划：

- [ ] 增加新手引导和三分钟试玩任务；
- [ ] 给每个指标增加“它测量什么、可能遗漏什么”的说明；
- [ ] 提供 20、45、90 分钟课堂模式；
- [ ] 增加结果导出或本地打印的讨论记录卡；
- [ ] 完善键盘操作、减少动态效果和屏幕阅读支持；
- [ ] 增加中英文基础术语表；
- [ ] 建立课堂反馈 Issue 模板和试用记录格式；
- [ ] 为浏览器核心评分补充自动化测试。

验收：

- 至少三名非开发者能独立完成主流程；
- 教师能在 45 分钟内完成“预测—操作—比较—复盘”；
- 关键结论不依赖颜色或音效；
- 教学者能清楚说明模型局限。

## 然后：v0.3 多角色课堂游戏 MVP

目标：把“调整滑块”升级为“在不同位置上作决定”。

最小范围：

- [ ] 六个角色：资本/投资者、平台经营团队、劳动者、商户、消费者、政府/监管者；
- [ ] 一局 4–6 轮，每轮 5–8 分钟；
- [ ] 共享仪表盘：平台盈余、履约质量、劳动健康、收入分配、商户压力、公众信任；
- [ ] 角色卡：公开目标、私有目标、资源和行动；
- [ ] 事件卡：暴雨、需求高峰、事故、商家延误、补贴竞争、监管检查；
- [ ] 谈判与规则调整阶段；
- [ ] 结局不是单一总分，而是多指标后果和复盘报告；
- [ ] 单设备传递模式，优先适配普通教室。

验收：

- 不同角色都能作出至少两种合理选择；
- 任一角色都不能单独控制全部结果；
- 学习者能指出至少一个反馈回路和一个风险转移；
- 一局可在 45–60 分钟内完成；
- 游戏记录不包含真实姓名或个人行为画像。

详细机制见 GAME_DESIGN.md。

## 中期：v0.4 教室会话

目标：支持教师组织多组同时体验和比较。

候选能力：

- 房间码与匿名小组；
- 教师控制台、回合计时和事件投放；
- 小组间政策与结果对照；
- 本地优先的数据保存与一键清除；
- 可打印角色卡和离线备选方案；
- 经过同意的匿名课堂反馈。

进入这一阶段前，必须先完成隐私威胁模型、未成年人使用说明、数据最小化方案和维护责任评估。不会为了“多人在线”而默认收集账号、聊天记录或精细行为数据。

## 长期：v1.0 开放课程平台

愿景：

- 按小学、中学、大学提供分层课程包；
- 支持外卖、网约车、内容推荐、直播和零工平台等案例；
- 情境与规则插件化；
- 教师可创建不含代码的自定义案例；
- 多语言和本地化制度背景；
- 研究资料与模型假设可追溯；
- 建立稳定的贡献者与教师共创社区。

## 跨版本工作

这些事项贯穿全部阶段：

- 可解释性：每个结果都能追到输入、约束和权重；
- 研究质量：现实性主张必须有来源，模拟假设必须标注；
- 公平与尊重：避免把劳动者或其他角色工具化、污名化；
- 无障碍：键盘、对比度、文字替代和低动态；
- 隐私：默认本地、最少数据、可清除；
- 可维护性：依赖最少、文档同步、小步发布；
- 课堂验证：真实观察学习者如何理解，而不只看功能是否运行。

## 建议优先创建的 Issue

1. 浏览器评分函数的单元测试与模块拆分；
2. 三分钟新手任务；
3. 五项指标的偏差与局限说明卡；
4. 45 分钟课堂试讲脚本；
5. 多角色游戏的纸面原型；
6. 第一轮教师与学生可用性测试。

路线图不是功能愿望清单。每个新功能应说明它帮助学习者看见了什么此前看不见的关系。
